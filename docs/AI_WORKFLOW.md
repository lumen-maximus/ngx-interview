# AI Workflow — Platform Ops Auditor

## Why this document exists

Platform Ops Auditor uses **GitHub Copilot Chat / Copilot cloud agent** as its AI-native development workflow. This document explains how AI was used, how human review was structured, and how the workflow could be improved.

## Tools used

- **GitHub Copilot Chat** — interactive code generation, refactoring, explanation
- **GitHub Copilot cloud agent** — autonomous PR generation from issue prompts
- **`.github/copilot-instructions.md`** — repository-level guardrails consumed by Copilot

## Initial PR

The initial implementation PR for this project was generated by the **Copilot cloud agent** from a structured prompt. It produced:

- The Lambda handler (`app/handler.py`)
- The Terraform configuration (originally a single `main.tf`, later refactored into modules)
- The Python unit-test suite (`app/test_handler.py`)
- The Terraform native test suite (`terraform/tests/platform_ops_auditor.tftest.hcl`)
- The CI/CD workflow (`.github/workflows/terraform.yml`)
- The README, DECISIONS.md, and architecture diagram

## Staged AI workflow

Iteration was deliberately **staged** rather than monolithic. Each stage has a clear human review focus:

1. **Generate MVP** — Copilot scaffolds the minimum viable architecture: handler, Terraform, tests, docs.
2. **Review and harden security** — Human review focuses on IAM hygiene (no `Action: *`, no `Resource: *`, no ARN wildcards), API permission scoping, and removal of the AWS managed `AWSLambdaBasicExecutionRole` (which introduces wildcard log-stream permissions).
3. **Strengthen operational intelligence** — `GET /summary` is enhanced with `top_findings` and `recent_operational_events`. Aggregation logic gets unit tests.
4. **Add optional AI maturity features** — Optional `POST /summarize` endpoint backed by Amazon Bedrock, gated by `enable_bedrock_summary` (default `false`). Refactor Terraform into modules.
5. **Final interview-readiness review** — Consistency check across code, infrastructure, tests, README, DECISIONS.md, diagram, CI.

## Where Copilot was course-corrected

Copilot generated reasonable starting points but required correction in several areas:

| Area | Default suggestion | Course correction |
|---|---|---|
| IAM | `AWSLambdaBasicExecutionRole` managed policy | Hand-crafted `aws_iam_policy_document` with exact table ARNs |
| CloudWatch Logs | `logs:CreateLogStream` on `arn:aws:logs:*:*:log-group:/aws/lambda/*:*` | Removed entirely; structured operational events stored in DynamoDB |
| API Gateway permissions | `source_arn = "arn:aws:execute-api:*:*:*"` | Exact `{stage}/{method}/{path}` ARNs per endpoint |
| Lambda zipping | `source_file = "handler.py"` | `source_dir = "../app"` to include all module files |
| Compute platform | Tendency to suggest ECS/Fargate "for production" | Kept Lambda for true serverless scale-to-zero |
| Database | Tendency to suggest RDS "for relational data" | Kept DynamoDB; the data model is document-shaped |
| Auth | Tendency to suggest Cognito | None — out of scope for an internal MVP |
| Bedrock | Initially scoped `bedrock:*` on `*` | Scoped to `bedrock:InvokeModel` on a single exact model ARN, behind a feature flag |
| Test mocks | Used real `aws_iam_policy_document` JSON in assertions | Switched to inspecting structured policy inputs for mock-provider compatibility |

## Future improvements to the AI workflow

- **AI-assisted PR review comments** — a Copilot reviewer that flags wildcard IAM, missing tests, and missing observability before human review.
- **Policy-as-code explanations** — Copilot generating natural-language summaries of every IAM policy change in PR descriptions.
- **Safe AI-generated operational summaries** — extending `POST /summarize` to push daily summaries into a Slack channel via SNS-fanout.
- **Stronger prompt context** — surfacing `DECISIONS.md` to Copilot as required reading so future generations don't re-suggest banned patterns.
- **Eval harness for Copilot output** — automated checks (CodeQL, `tflint`, `checkov`) gating Copilot-generated diffs in CI.
